# 功能规格说明：历史高低价

**功能分支**: `master`  
**创建日期**: 2026-03-28  
**状态**: 已实现  
**输入**: 用户描述：「新增功能，股票的历史最高价和历史最低价：从数据库中的日股价中通过定时任务计算每个股票的历史最高价和历史最低价，用于后面的选股策略分析中。」

## Clarifications

### Session 2026-03-28

- Q: 手动全量重算是否提供 HTTP 接口、如何鉴权？ → A: **不对外暴露 HTTP**；全量重算在**本机**通过 **CLI / 运维脚本**执行即可，无需提供独立 HTTP 接口。

## 用户场景与测试 *（必填）*

### 用户需求 1 - 每只股票具备可复用的历史最高价与历史最低价（优先级：P1）

作为分析与选股策略的输入，系统需要为**每一只有日行情记录的股票**维护「历史最高价」与「历史最低价」，其含义与口径与**已落库的日行情序列**一致；数值**持久化在股票基础信息表 `stock_basic`**（与主数据同行），供后续策略模块引用。

**为何此优先级**：没有统一、可追溯的历史高低价，后续选股策略无法在同一数据口径下比较或筛选。

**独立测试**：在已知日行情样本下，核对输出与按业务口径手工汇总的结果一致，即可验证本需求，无需依赖具体策略功能是否已实现。

**验收场景**：

1. **前置条件** 某股票在库中已存在多条按交易日排列的日行情记录，**当** 系统完成该股票的历史高低价计算，**那么** 历史最高价应等于该股票在**全部已存日行情**中「当日最高价」字段的**最大值**，历史最低价应等于「当日最低价」字段的**最小值**（字段语义以项目日行情模型为准）。
2. **前置条件** 某股票仅有单日行情，**当** 完成计算，**那么** 历史最高价与历史最低价应分别等于该日的最高价与最低价。

---

### 用户需求 2 - 增量定时更新与手动全量重算（优先级：P2）

作为运维与数据消费者，我希望采用**两套互补**的更新方式：（1）**增量**定时任务——在日行情已入库后的较晚时段自动刷新极值（默认每个交易日约 **18:00**，具体时区与配置以部署为准），日常运行成本低；（2）**全量**重算——由运维在**本机**通过 **CLI / 脚本手动触发**（不提供对外 HTTP），用于补数、历史日行情大面积修订后与「全量口径」对齐等场景。

**为何此优先级**：历史极值相对日行情明细变更频率低，放在晚间增量即可满足多数策略时效；全量手动保证在异常修订下仍可纠偏。

**独立测试**：增量路径在新增一日行情后经一次定时窗口验证；全量路径在本机执行 CLI/脚本后验证全市场或约定范围与手工验算一致。

**验收场景**：

1. **前置条件** 增量调度已启用，且当日日行情已写入，**当** 到达约定的增量执行时刻（如 **18:00**），**那么** 受影响股票在 `stock_basic` 中的历史最高价、历史最低价应已按「用户需求 1」的口径更新（在增量算法覆盖范围内）。
2. **前置条件** 运维或授权人员需要全量纠偏，**当** 在**本机**执行一次**手动全量**重算（CLI/脚本），**那么** 全表或约定范围内股票的历史极值应与基于**当前库中日行情全集**的汇总结果一致。
3. **前置条件** 某一执行周期内没有新的日行情写入，**当** 增量任务仍执行，**那么** 系统行为应可预期（不破坏已有正确结果），且可留下可理解的执行摘要。

---

### 用户需求 3 - 下游策略可依赖的数据一致性与可核对性（优先级：P3）

作为后续选股或分析功能的设计者，我希望在相同日行情快照下，多次读取到的历史最高价、历史最低价**一致**，且能通过日行情数据**核对**来源，避免「黑箱数字」。

**为何此优先级**：策略回测与实盘对比要求可复现；不可核对则难以排错。

**独立测试**：固定日行情数据集，多次查询或多次任务运行后比对结果是否相同；必要时用导出或查询日行情自行验算。

**验收场景**：

1. **前置条件** 日行情数据未发生变化，**当** 在不同时间或多次触发更新后读取同一股票的历史高低价，**那么** 结果应保持一致。
2. **前置条件** 需要排查某一股票指标是否合理，**当** 业务方依据日行情按约定口径验算，**那么** 验算结果应与系统给出的历史最高价、历史最低价相符。

---

### 用户需求 4 - 股票基本信息页列表展示历史极值（优先级：P2）

作为用户，我希望在**股票基本信息**页面的**分页列表**中直接看到每只股票的**历史最高价**与**历史最低价**（与「用户需求 1」同一口径），无需打开其它页面，便于与代码、名称、交易所等信息对照查看。

**为何此优先级**：列表是确认主数据是否可用的主入口；极值与基础信息同屏可减少跳转，并为后续人工核对日行情提供入口感。

**独立测试**：仅打开股票基本信息页，检查列表列是否出现两列且数值与 `stock_basic` / 接口返回一致；无数据时展示符合边界约定。

**验收场景**：

1. **前置条件** 某股票已计算并写入历史极值，**当** 用户在股票基本信息页浏览列表（含分页），**那么** 该行应展示**历史最高价**、**历史最低价**两列，数值与后台存储及「用户需求 1」口径一致。
2. **前置条件** 某股票尚无日行情或极值未计算，**当** 用户查看列表，**那么** 两列应以约定方式表示缺数（如「—」或空白），**不得**显示无意义的 `0` 冒充有效价格。

---

### 边界情况

- 当某股票尚无任何日行情时，`stock_basic` 中对应极值字段应以明确约定的方式表示缺数（如 `NULL`），**不**用 0 冒充有效极值。
- 当某日行情发生**更正**或**历史补录**导致仅靠「自上次增量以来」无法覆盖时，应通过**手动全量**重算使 `stock_basic` 与**当前库中日行情全集**一致；增量任务可仅保证常规「新增交易日」场景下的时效与性能。
- 当全市场股票数量很大时，单次自动更新应在可接受的业务时限内完成或可分解为可观测的批次（具体时限见成功标准），不应长期停留在「部分股票已更新、部分未更新」且无法区分。
- 当自动更新部分失败时，系统应能区分「整体成功」「部分失败需重试」等等价可运维状态，并避免未说明的半更新导致策略误用（失败时保留旧极值等约定见假设）。
- 列表展示时若列宽有限，历史极值列仍应可读（合理精度或格式化），且与 **003-股票基本信息** 页面其它列风格一致。

## 功能需求 *（必填）*

### 功能要求

- **FR-001**: 系统必须能基于**已存储的日行情数据**，为每只有效日行情的股票计算**历史最高价**（全部历史日中「当日最高价」的最大值）与**历史最低价**（全部历史日中「当日最低价」的最小值），口径与日行情字段定义一致。
- **FR-002**: 系统必须将上述结果**写入并维护于 `stock_basic`**，与股票基础主数据**一行对应**（每代码一条），便于与名称、交易所等一并查询。
- **FR-003**: 系统必须提供**增量**定时更新能力，默认在每个交易日约 **18:00** 执行（具体时间可配置），在日行情入库后对受影响标的刷新极值。
- **FR-004**: 系统必须提供**手动触发的全量**重算能力（通过**本机 CLI 或运维脚本**调用同一套计算逻辑，**不**提供对外 HTTP 接口），使在需要时能以「对日行情全集重算」的口径覆盖全部或约定范围，并与增量路径在文档中区分适用场景。
- **FR-005**: 在日行情数据集不变的前提下，同一股票的历史最高价、历史最低价必须**可重复、可核对**（与手工按同一口径验算一致）。
- **FR-006**: 系统必须为每次增量或全量任务提供**可观测结果**（例如成功/失败、影响范围或条数类摘要），便于运维与排错，不向最终用户暴露内部实现细节或原始堆栈。
- **FR-007**: **股票基本信息**页面（见 **003-股票基本信息**）的分页列表必须展示**历史最高价**、**历史最低价**两列，数据与 `stock_basic` 存储及本规格口径一致；列表接口须在每条记录中返回对应字段。
- **FR-008**: 其它功能（如选股、分析页）若展示历史极值，应引用**同一套** `stock_basic` 字段与同一口径，避免重复计算或不一致。

### 关键实体 *（若涉及数据则必填）*

- **日行情**：表示某股票在某一交易日的官方或数据源约定的开高低收、成交量等序列数据；本功能仅依赖其中的**最高价、最低价**及交易日标识，用于汇总历史极值。
- **股票历史高低价**：表示某股票在**当前库中全部历史日行情**范围内汇总得到的两个标量：历史最高价、历史最低价；**持久化字段位于 `stock_basic`**，与单只证券的基础信息同行存储；通过**增量定时任务**与**手动全量**两种路径与日行情保持同步（见功能要求）。

## 成功标准 *（必填）*

### 可衡量结果

- **SC-001**: 在常规数据规模与正常运行下，日行情入库后，经一次**增量**执行窗口（默认不晚于当日约 **18:00** 后的首次成功执行），**绝大多数**受影响股票在 `stock_basic` 中的历史高低价与最新日行情一致（可通过抽检与日行情核对验证）；手动全量执行后，约定范围内股票应与全量口径一致。
- **SC-002**: 对固定日行情快照，**连续两次**读取同一股票的历史最高价、历史最低价，结果**完全一致**。
- **SC-003**: 运维或数据负责人在一次**增量或全量**任务执行后，能在**5 分钟内**从系统提供的摘要中判断本轮是否成功、是否存在需处理的失败（无需阅读代码或原始日志文件即可完成判断）。
- **SC-004**: 在后续选股或分析功能引用本数据时，**不因**历史高低价口径不明或与日行情矛盾而导致策略结论无法向业务方解释（可通过评审或联调 checklist 验证）。
- **SC-005**: 用户在股票基本信息页打开列表后，**首屏可见行**中应能直接看到**历史最高价**、**历史最低价**列（与代码、名称等并列），无需额外展开或跳转；无数据时呈现符合验收场景约定的缺省展示。

## 假设

- 「日股价」与项目现有**日行情**（如 `stock_daily_bar`）一致；最高价、最低价字段含义与数据源文档及项目既有日行情模型一致。
- **历史**指当前数据库中该股票**全部已存储交易日**的集合；若未来需「近 N 年」等滚动窗口，由后续规格扩展，本功能首版以**全历史**为准。
- **持久化位置**：历史最高价、历史最低价作为**准静态指标**落在 **`stock_basic`**，与股票基础信息同表维护，读策略与选股分析时与代码、名称等一并获取即可。
- **更新策略**：**增量**定时任务在日行情同步完成后的较晚时段执行，默认**每个交易日约 18:00**（本地/业务时区可配置），满足「不必紧跟分钟级行情」的假设；**全量**重算由运维在**本机**通过 **CLI/脚本手动触发**，**不**提供全量重算的对外 HTTP 接口。
- **任务失败**：增量或全量执行失败时，应保留上一轮已成功写入的极值（避免误写空值），并在日志或摘要中记录失败范围，便于重试；不强制自动回滚数据库事务以外的业务状态（见边界情况）。
- 无日行情的股票在 `stock_basic` 中对应极值字段为**空或未计算**等约定状态，**不**用 0 冒充有效极值。
- 本功能聚焦**计算与持久化、两类更新路径、可观测性**，以及**股票基本信息页列表展示**（归属 **003** 页面范围，数据口径由本规格定义）；不包含具体选股策略逻辑，策略模块从 `stock_basic` 或项目统一数据接口读取即可。
- **003-股票基本信息** 页面须在列表中增加历史最高价、最低价列；列旁宜提供简短**悬浮说明**（如 Tooltip），说明「基于已入库日线的历史极值、非当日盘中价」，避免与实时行情混淆（实现细节见前端规范）。
- 权限、多租户、对外公开 API 形态若项目另有全局规格，本功能遵循项目统一约定。
