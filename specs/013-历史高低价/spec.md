# 功能规格说明：历史高低价

**功能分支**: `master`  
**创建日期**: 2026-03-28  
**状态**: 已实现（2026-03-29 修订：极值**按交易日**落在 `stock_daily_bar`，避免回测用 `stock_basic` 全表终态泄露未来信息）  
**输入**: 用户描述：「新增功能，股票的历史最高价和历史最低价：从数据库中的日股价中通过定时任务计算每个股票的历史最高价和历史最低价，用于后面的选股策略分析中。」

## Clarifications

### Session 2026-03-28

- Q: 手动全量重算是否提供 HTTP 接口、如何鉴权？ → A: **不对外暴露 HTTP**；全量重算在**本机**通过 **CLI / 运维脚本**执行即可，无需提供独立 HTTP 接口。

## 用户场景与测试 *（必填）*

### 用户需求 1 - 每只股票具备可复用的历史最高价与历史最低价（优先级：P1）

作为分析与选股策略的输入，系统需要为**每一只有日行情记录的股票**、**每一个交易日**维护「截至该日（含）的累计历史最高价」与「累计历史最低价」（对 `high`/`low` 做时间顺序上的扩展最大/最小，跳过 `NULL` 参与比较）；数值**持久化在 `stock_daily_bar`** 的 `cum_hist_high` / `cum_hist_low`，**不得**仅在 `stock_basic` 存全表终态极值，以免回测在历史上的某日误用未来才出现的高低点。列表页展示的「历史最高/最低」可取**最新一根日线**上的累计列，与全表 `MAX/MIN` 在数据完整时一致，但**回测与策略在日期 T 必须读取 `trade_date=T` 的日线行**。

**为何此优先级**：没有统一、可追溯的历史高低价，后续选股策略无法在同一数据口径下比较或筛选。

**独立测试**：在已知日行情样本下，核对输出与按业务口径手工汇总的结果一致，即可验证本需求，无需依赖具体策略功能是否已实现。

**验收场景**：

1. **前置条件** 某股票在库中已存在多条按交易日排列的日行情记录，**当** 系统完成累计极值计算，**那么** 在任意交易日 `T` 的日线行上，`cum_hist_high` 应等于 `trade_date ≤ T` 范围内「当日最高价」的**最大值**，`cum_hist_low` 应等于该范围内「当日最低价」的**最小值**（`NULL` 不参与比较，语义以项目日行情模型为准）。
2. **前置条件** 某股票仅有单日行情，**当** 完成计算，**那么** 该行的 `cum_hist_high`/`cum_hist_low` 应分别等于该日非空的最高价/最低价（若当日 high/low 为空则累计列可为 `NULL` 直至出现有效价）。

---

### 用户需求 2 - 随日线写入的增量递推与手动全量重算（优先级：P2）

作为运维与数据消费者，我希望采用**两套互补**的更新方式：（1）**增量**——在每次日行情 **upsert 成功并落库可见**后，对受影响行用「上一交易日累计值 + 当日 high/low」做 O(1) 递推；若写入落在历史中间（存在更晚 `trade_date`）或检测到累计列未回填链，则对该股**整段序列**重算，**无需**单独 Cron；（2）**全量**重算——由运维在**本机**通过 **CLI / 脚本手动触发**（不提供对外 HTTP），用于补数、历史日行情大面积修订后与「全量口径」对齐等场景。

**为何此优先级**：极值与日行情强一致、延迟最低；全量手动保证在异常修订下仍可纠偏。

**独立测试**：写入新一日行情后同事务/同批内 `cum_hist_*` 已正确；全量路径在本机执行 CLI/脚本后验证全市场或约定范围与手工验算一致。

**验收场景**：

1. **前置条件** 当日日行情同步任务已写入（或回灌写入）一行新日线，**当** 该次写入流程结束，**那么** 该股该行及后续一致性约束下的 `cum_hist_*` 应已按「用户需求 1」与当前库中日行情一致（纯尾部追加为 O(1) 递推）。
2. **前置条件** 运维或授权人员需要全量纠偏，**当** 在**本机**执行一次**手动全量**重算（CLI/脚本），**那么** 全市场有日线股票的每一行累计极值应与基于**当前库中日行情全集**按日递推结果一致。
3. **前置条件** 某一周期内没有新的日行情写入，**那么** 不应依赖「空跑」定时任务维持正确性；已有累计列在无新写入时保持不动即可。

---

### 用户需求 3 - 下游策略可依赖的数据一致性与可核对性（优先级：P3）

作为后续选股或分析功能的设计者，我希望在相同日行情快照下，多次读取到的历史最高价、历史最低价**一致**，且能通过日行情数据**核对**来源，避免「黑箱数字」。

**为何此优先级**：策略回测与实盘对比要求可复现；不可核对则难以排错。

**独立测试**：固定日行情数据集，多次查询或多次任务运行后比对结果是否相同；必要时用导出或查询日行情自行验算。

**验收场景**：

1. **前置条件** 日行情数据未发生变化，**当** 在不同时间或多次触发更新后读取同一股票**同一交易日**日线上的 `cum_hist_*`，**那么** 结果应保持一致。
2. **前置条件** 需要排查某一股票指标是否合理，**当** 业务方依据日行情按约定口径验算，**那么** 验算结果应与系统在该日日线行上给出的累计极值相符；**最新交易日**行的累计值应与全表 `MAX(high)`/`MIN(low)` 一致（在 high/low 覆盖正常时）。

---

### 用户需求 4 - 股票基本信息页列表展示历史极值（优先级：P2）

作为用户，我希望在**股票基本信息**页面的**分页列表**中直接看到每只股票的**历史最高价**与**历史最低价**（与「用户需求 1」同一口径），无需打开其它页面，便于与代码、名称、交易所等信息对照查看。

**为何此优先级**：列表是确认主数据是否可用的主入口；极值与基础信息同屏可减少跳转，并为后续人工核对日行情提供入口感。

**独立测试**：仅打开股票基本信息页，检查列表列是否出现两列且数值与**最新日线** `cum_hist_*` / 接口返回一致；无数据时展示符合边界约定。

**验收场景**：

1. **前置条件** 某股票已计算并写入日线累计极值，**当** 用户在股票基本信息页浏览列表（含分页），**那么** 该行应展示**历史最高价**、**历史最低价**两列，数值与**该股票最新交易日**日线行上的 `cum_hist_high`/`cum_hist_low` 一致（接口字段名可仍为 `hist_high`/`hist_low` 以保持兼容）。
2. **前置条件** 某股票尚无日行情或极值未计算，**当** 用户查看列表，**那么** 两列应以约定方式表示缺数（如「—」或空白），**不得**显示无意义的 `0` 冒充有效价格。

---

### 边界情况

- 当某股票尚无任何日行情时，列表接口中对应极值字段为 `null`；日线表无行可写。
- 当某日行情发生**更正**或**历史补录**时：若写入落在存在更晚交易日的位置，同步流程会对该股**整股重算**累计列；若仍不放心或批量修订，应通过**手动全量**重算使该股（或全市场）与**当前库中日行情全集**递推一致。
- 当全市场股票数量很大时，单次自动更新应在可接受的业务时限内完成或可分解为可观测的批次（具体时限见成功标准），不应长期停留在「部分股票已更新、部分未更新」且无法区分。
- 当自动更新部分失败时，系统应能区分「整体成功」「部分失败需重试」等等价可运维状态，并避免未说明的半更新导致策略误用（失败时保留旧极值等约定见假设）。
- 列表展示时若列宽有限，历史极值列仍应可读（合理精度或格式化），且与 **003-股票基本信息** 页面其它列风格一致。

## 功能需求 *（必填）*

### 功能要求

- **FR-001**: 系统必须能基于**已存储的日行情数据**，为每只股票、**按交易日升序**计算并写入**截至该日（含）的累计**最高价（扩展 `MAX(high)`）与累计最低价（扩展 `MIN(low)`），口径与日行情 `high`/`low` 一致；`NULL` 的 high/low 不参与当步比较，累计列继承此前有效扩展值。
- **FR-002**: 系统必须将上述结果**写入并维护于 `stock_daily_bar`** 的 `cum_hist_high`、`cum_hist_low`，**禁止**将「全表终态」极值作为策略/回测在历史上的唯一来源存放在 `stock_basic`（`stock_basic` 可不存或使用列表展示用的派生逻辑）。
- **FR-003**: 系统必须在**日行情写入路径**内维护 `cum_hist_*`（尾部追加 O(1) 递推；中间改数触发该股整段重算），**不**依赖单独定时 Job 完成日常增量。
- **FR-004**: 系统必须提供**手动触发的全量**重算能力（通过**本机 CLI 或运维脚本**调用同一套计算逻辑，**不**提供对外 HTTP 接口），使在需要时能以「对日行情全集重算」的口径覆盖全部或约定范围，并与增量路径在文档中区分适用场景。
- **FR-005**: 在日行情数据集不变的前提下，同一股票的历史最高价、历史最低价必须**可重复、可核对**（与手工按同一口径验算一致）。
- **FR-006**: 系统必须为每次增量或全量任务提供**可观测结果**（例如成功/失败、影响范围或条数类摘要），便于运维与排错，不向最终用户暴露内部实现细节或原始堆栈。
- **FR-007**: **股票基本信息**页面（见 **003-股票基本信息**）的分页列表必须展示**历史最高价**、**历史最低价**两列，数值等于该股票**最新交易日**日线上的 `cum_hist_high`/`cum_hist_low`；列表接口可仍返回 `hist_high`/`hist_low` 字段名以保持兼容。
- **FR-008**: **回测与选股策略**在模拟日 `T` 必须使用 **`trade_date = T` 的日线行**读取累计极值（或等价地仅用 `T` 及之前数据现场计算），**禁止**使用 `stock_basic` 上曾存在的全表终态极值作为历史某日输入。

### 关键实体 *（若涉及数据则必填）*

- **日行情**：表示某股票在某一交易日的官方或数据源约定的开高低收、成交量等序列数据；本功能仅依赖其中的**最高价、最低价**及交易日标识，用于汇总历史极值。
- **股票历史高低价（按日）**：在每个 `trade_date` 上表示「截至该日（含）」的扩展最高/最低；**持久化于 `stock_daily_bar.cum_hist_high` / `cum_hist_low`**；通过**日线写入时递推**与**手动全量**两种路径与日行情保持同步。列表展示的「全历史」观感由**最新日线行**的累计列体现。

## 成功标准 *（必填）*

### 可衡量结果

- **SC-001**: 在常规数据规模与正常运行下，日行情**写入流程完成后**，当日有日线的股票其受影响行的 `cum_hist_*` 与递推口径一致；手动全量执行后，全市场应与全量递推一致。
- **SC-002**: 对固定日行情快照，**连续两次**读取同一股票的历史最高价、历史最低价，结果**完全一致**。
- **SC-003**: 运维或数据负责人在一次**增量或全量**任务执行后，能在**5 分钟内**从系统提供的摘要中判断本轮是否成功、是否存在需处理的失败（无需阅读代码或原始日志文件即可完成判断）。
- **SC-004**: 在后续选股或分析功能引用本数据时，**不因**历史高低价口径不明或与日行情矛盾而导致策略结论无法向业务方解释（可通过评审或联调 checklist 验证）。
- **SC-005**: 用户在股票基本信息页打开列表后，**首屏可见行**中应能直接看到**历史最高价**、**历史最低价**列（与代码、名称等并列），无需额外展开或跳转；无数据时呈现符合验收场景约定的缺省展示。

## 假设

- 「日股价」与项目现有**日行情**（如 `stock_daily_bar`）一致；最高价、最低价字段含义与数据源文档及项目既有日行情模型一致。
- **历史**指当前数据库中该股票**全部已存储交易日**的集合；若未来需「近 N 年」等滚动窗口，由后续规格扩展，本功能首版以**全历史**为准。
- **持久化位置**：累计历史极值落在 **`stock_daily_bar`**；**回测按日读取**；列表可与基础信息一并查询时通过**最新日线**左连或子查询得到展示用 `hist_high`/`hist_low`。
- **更新策略**：**增量**与日行情 **upsert** 同路径完成（见 FR-003）；**全量**重算由运维在**本机**通过 **CLI/脚本手动触发**，**不**提供全量重算的对外 HTTP 接口。
- **任务失败**：增量或全量执行失败时，应保留上一轮已成功写入的极值（避免误写空值），并在日志或摘要中记录失败范围，便于重试；不强制自动回滚数据库事务以外的业务状态（见边界情况）。
- 无日行情的股票在列表接口中极值字段为 **null**，**不**用 0 冒充有效极值。
- 本功能聚焦**按日累计极值的计算与持久化、两类更新路径、可观测性**，以及**股票基本信息页列表展示**（归属 **003** 页面范围）；策略/回测须按交易日读日线表或等价接口。
- **003-股票基本信息** 页面须在列表中增加历史最高价、最低价列；列旁宜提供简短**悬浮说明**（如 Tooltip），说明「基于已入库日线的历史极值、非当日盘中价」，避免与实时行情混淆（实现细节见前端规范）。
- 权限、多租户、对外公开 API 形态若项目另有全局规格，本功能遵循项目统一约定。
