# 调研结论：历史高低价（Phase 0）

**关联规格**：[spec.md](./spec.md)  
**日期**：2026-03-28

## 1. 增量任务与日线同步的先后关系

- **决策**：增量极值任务安排在 **17:00 股票同步之后**，采用 **每个交易日 18:00（Asia/Shanghai）** 的独立 Cron 任务；与现有 `app.core.scheduler` 中 **17:00** `_job_sync_stock`、**17:20** 策略任务错开，给日线入库留出时间窗口。
- **理由**：规格要求「日行情入库后再刷新极值」；当前编排下日线由 17:00 批次驱动，18:00 执行可显著降低「日线尚未写完就算极值」的概率。若个别日 17:00 任务耗时过长跨至 18:00 之后，仍可能读到未完整数据——此类情况依赖次日增量或**手动全量 CLI** 纠偏（规格已允许）。
- **备选**：在 `execute_pending_auto_sync` 成功回调末尾联动调用极值更新——耦合度高、需改 orchestrator，且失败重试边界复杂，故不采用为首版方案。

## 2. 「增量」计算口径（实现策略）

- **决策**：对每个目标交易日 `T`（通常等于当日且为开市日），仅对「在 `stock_daily_bar` 中存在 `trade_date = T` 的记录的 `stock_code`」集合，执行该股在**全库日线**上的 `MAX(high)` / `MIN(low)` 并写回 `stock_basic`；**不**采用「旧极值与新一日 high/low 比大小」的纯增量公式，以免历史修订后错误。
- **理由**：与规格「全历史口径」一致；仅缩小**参与重算的股票集合**，控制 18:00 任务耗时，仍保证单股结果与全量聚合一致。
- **备选**：每晚全市场 `GROUP BY stock_code` 一次大 SQL 更新——实现简单但全市场扫描压力大；可作为 CLI 全量或运维低谷期策略保留。

## 3. 全量重算入口形态

- **决策**：**不提供** HTTP；在仓库内提供 **本机可执行脚本**（如 `backend/scripts/recompute_hist_extrema_full.py`），内建 `SessionLocal` + 服务层 `run_full_recompute`，支持从项目根或 `backend` 目录以 `python scripts/...` 调用。
- **理由**：与 Clarifications（用户明确本机 CLI、不对外暴露）一致；与现有「无统一 CLI 模块、以脚本为主」的仓库习惯兼容。
- **备选**：`manage.py` 子命令——需新建基础设施，非必要。

## 4. 复权与数据源

- **决策**：极值直接取自 `stock_daily_bar.high` / `low` 已存数值，**不**在任务内另行做复权转换；与项目当前日线表语义一致。
- **理由**：规格假设「日行情模型一致」；若未来日线改为多复权版本，应通过日线表/schema 演进单独规格，而非本任务内猜测。
- **备选**：单独存不复权/前复权两套极值——超出当前规格范围。

## 5. 与 003 列表 API 的契约关系

- **决策**：列表仍使用 **`GET /api/stock/basic`**，在 `items[]` 中**扩展字段** `hist_high`、`hist_low`、`hist_extrema_computed_at`；不在 013 新增独立 REST 资源路径。
- **理由**：数据落在 `stock_basic`，与 003 页面一一对应；减少前端请求数。
- **备选**：单独 `GET /stock/basic/extrema`——多一次请求，不必要。
