# 调研结论：底部盘整突破策略

**日期**：2026-03-30  
**功能规格**：`specs/017-底部盘整突破/spec.md`

## 技术选型

### 1. 策略框架复用

**决策**：复用现有 `StockStrategy` Protocol 和回测引擎

**理由**：
- 项目已有成熟的策略框架（`strategy_base.py`），包含 `execute()` 和 `backtest()` 方法
- 现有策略（冲高回落战法、恐慌回落法）已验证框架可行性
- 复用可确保不同策略的回测结果可比、API 一致

**备选**：独立实现策略逻辑
- **为何不采用**：会增加维护成本，与现有系统不一致

### 2. 历史最高价数据源

**决策**：使用 `stock_daily_bar.cum_hist_high` 字段

**理由**：
- 该字段已在 013-历史高低价 中实现，记录截至当日的历史最高价
- 每日递推更新，无需额外计算
- 回测时使用当日行的值，保证"当时已知"的正确性

**备选**：实时计算历史最高价
- **为何不采用**：性能开销大，且与已有数据口径不一致

### 3. 盘整天数计算方式

**决策**：仅计算交易日，非交易日不计入

**理由**：
- 规格明确要求"15个交易日"
- 数据库中的日线数据仅包含交易日
- 符合股市实际情况

**备选**：按自然日计算
- **为何不采用**：与规格不符，且周末无交易数据

### 4. 条件单模拟方式

**决策**：使用日线最低价判断触碰，以触发价成交

**理由**：
- 规格澄清中确认采用"条件单"模式
- 日线最低价代表当日价格的最低点，可判断是否触及触发价
- 以触发价成交模拟了条件单的"触及即成交"特性

**备选**：使用收盘价判断
- **为何不采用**：无法反映盘中触碰触发价的情况

### 5. 止盈监控实现

**决策**：动态追踪持仓期间的最高收盘价

**理由**：
- 规格澄清中确认为动态追踪模式
- 每日收盘后检查是否创新高，更新最高价基准
- 从最高价回落 5% 时触发止盈

**备选**：使用固定止盈价
- **为何不采用**：与规格"动态追踪"要求不符

## 算法设计

### 盘整形态识别算法

**核心思路**：状态机模式，维护当前盘整状态

```
状态：无盘整 → 开始盘整（设置基准价）→ 继续盘整/突破/失效
转换条件：
  - 无盘整 + 收盘价 <= 历史最高价/2 → 开始盘整
  - 盘整中 + 偏离 <= 3% → 继续盘整
  - 盘整中 + 偏离 > 3% 且 天数 >= 15 → 突破信号
  - 盘整中 + 偏离 < -3% → 失效，重置
```

**时间复杂度**：O(n)，n 为日线数量

### 持仓监控算法

**核心思路**：逐日检查，维护状态

```
每日检查：
1. 更新最高收盘价
2. 计算收益率，若 >= 15% 则进入止盈监控
3. 检查止损条件：最低价 <= 止损价
4. 若在止盈监控中，检查止盈条件：最低价 <= 最高价 * 0.95
5. 优先级：止损 > 止盈
```

**时间复杂度**：O(m)，m 为持仓天数

## 性能考量

### 数据查询优化

**问题**：全 A 股近三年回测需查询约 5000 股 × 700 日 = 350 万条日线记录

**解决方案**：
1. 单次 SQL 批量查询，避免循环查询
2. 按股票代码分组，内存中处理
3. 使用索引：`(stock_code, trade_date)`

**预期性能**：
- 数据查询：约 10 秒
- 策略计算：约 60 秒
- 总耗时：约 1-2 分钟（满足 5 分钟目标）

## 边界情况处理

| 边界情况 | 处理方式 |
|---------|---------|
| 股票新上市，历史数据不足 60 天 | 跳过该股票，不纳入筛选 |
| 盘整期间停牌 | 停牌期间不计入盘整天数，复牌后继续 |
| 历史最高价字段为空 | 跳过该股票，记录日志 |
| 买入日/卖出日数据缺失 | 该笔交易标记为 skipped，不计入统计 |
| 止盈止损同时触发 | 优先执行止损，按止损价成交 |
