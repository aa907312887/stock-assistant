# Phase 0 调研结论：首页投资逻辑（升级）

## 1. 存储结构

- **决策**：采用**独立表** `investment_logic_entry`，每行一条历史；**技术面 / 基本面 / 消息面**各占一列（`TEXT`），权重三列**整数百分比**，**和为 100**；另设 **`extra_json`（JSON NULL）** 作扩展字段，与全库「结构化 + `extra_json`」策略一致。  
- **理由**：与规格升级一致；支持多条历史、按时间回顾；与 `user` 1:N；后续非核心字段可先落 JSON 再结构化迁移。  
- **备选**：`user` 表 JSON 列存数组。不采用：查询与校验不便、索引与约束弱。

## 2. 「当前」与登录响应

- **决策**：**当前**由服务端按 `user_id` 下 **`updated_at` 最大**（并列取 `id` 最大）解析；提供 **`GET /api/investment-logic/current`**；**不在** `UserOut` 中嵌入三面长文。  
- **理由**：减少登录与 `/me` 体积；首页按需拉取即可。  
- **备选**：扩展 `UserOut`。可作为等价实现，本计划以独立资源为准（见 `contracts/`）。

## 3. 新增与修改的时间语义

- **决策**：**新增**写 `created_at`；**修改**更新 `updated_at` 且不改 `created_at`。  
- **理由**：满足「每次新增记录时间」与历史回顾；修改不产生新行（除非用户走「新增一条」流程）。

## 4. 权重交互

- **决策**：存**整数百分比**，校验 **三者之和 = 100**；前端可用三个输入框或滑块归一化后提交。  
- **理由**：规则清晰、易测；与 SC-004 一致。

## 5. 迁移

- **决策**：手写 **CREATE TABLE**；若曾加 `user.investment_logic`，数据迁移脚本将旧单列文本**导入一条默认三面**（可全部落入「技术面」或要求用户重填，实现时选一种并写进迁移说明）。  
- **理由**：与仓库无 Alembic 的现状一致。

## 6. 定时任务

- **决策**：不涉及。
