# 调研记录：策略选股（冲高回落战法）

## 1. 策略表达与交付形态

- **决策**：策略以**代码实现**交付；每个策略有独立页面；本期仅交付“冲高回落战法”。
- **理由**：用户明确要求策略不可由界面配置，且每个策略都需要独立开发与演进；代码交付更适合复杂交易理念。
- **备选**：
  - **界面配置策略**：不符合需求；且会引入表达式解析、权限与安全问题，范围过大。

## 2. “为未来回测方便”的数据结构选择

- **决策**：本期即落库“策略执行快照 + 信号事件 + 候选明细”，并强制记录策略版本与执行口径，用于后续“回顾当时的操作现场”与回测复现。
- **理由**：回测本质上需要“可回放”的执行输入与信号序列；如果本期只返回前端列表而不沉淀执行记录，未来回测会被迫重构接口与数据模型。
- **备选**：
  - **不落库，仅返回结果**：短期快，但未来回测需要补齐历史执行记录，且难以解释“同一策略为何结果变了”。
  - **只落库候选列表，不落库信号事件**：回测需要入场/离场事件，候选列表不足以复现交易。

## 3. 冲高回落战法关键口径（无分时数据）

### 3.0 “大涨（大阳线）”的定义

- **决策**：用“当日从开盘价到最高价的最大涨幅”定义大阳线：\((high - open) / open \ge 10%\)。
- **理由**：你明确要求“大涨不能只看收盘涨幅”，必须体现盘中从开盘到最高点的强势拉升。
- **备选**：
  - 用 `pct_change`（收盘相对前收涨幅）阈值：无法体现盘中最高点强度，且会把“冲高回落但收盘涨幅不大”的情况漏掉。

### 3.1 回落比例的定义

- **决策**：用 \((high - close) / high\) 作为“从最高点回落比例”，阈值 3%。
- **理由**：与“从最高点回落”语义最一致；不依赖前收盘价；对不同涨幅幅度更直观。
- **备选**：
  - \((high - close) / prev_close\)：可读性较强，但更接近“相对前收的回落”，与“最高点回落”不完全一致。

### 3.2 “低开 3%”的定义

- **决策**：用 \((open_1 - close_0) / close_0\) 或 \((open_1 - prev_close_0) / prev_close_0\) 取其一，并在执行快照中固化口径字段；本期优先使用 `prev_close_0`（若存在），否则退化为 `close_0`。
- **理由**：多数行情口径“低开”相对前收；但数据完整性需考虑字段为空的情况。
- **备选**：
  - 始终用 `close_0`：简单，但与常用“前收”口径略有差异。

### 3.3 “开盘买入/卖出”的落地

- **决策**：本期信号事件仅记录“在开盘执行”的语义与对应交易日，价格层面不做真实成交价计算；未来接入分时数据后再扩展事件的成交价/滑点等字段。
- **理由**：当前无分时数据，强行计算会产生误导；但记录事件时间点与语义能满足未来回测复用。
- **备选**：
  - 用日线开盘价当作成交价：可做“粗回测”，但本期明确不做回测，为避免误导先不提供收益计算。

## 4. 接口形态（与综合选股区分）

- **决策**：新增 `GET/POST /api/strategies/...` 一组接口，与现有综合选股 `GET /api/stock/screening` 分离。
- **理由**：综合选股是“条件筛选列表”，策略选股是“策略执行 + 信号事件 + 执行快照”，语义不同且未来演进不同。
- **备选**：
  - 复用 `stock/screening`：会把策略能力挤入现有筛选参数体系，后期难维护。
