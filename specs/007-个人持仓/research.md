# 调研结论：个人持仓（007）

**功能**: 007-个人持仓 | **日期**: 2026-03-22

本文档收敛 Phase 0 待澄清项，供 `plan.md` 与 `data-model.md` 直接采用。

## 1. 参考市值价格来源

- **决策**：当前持仓的**参考价**取 `stock_daily_bar` 中该 `stock_code` **最新 `trade_date`** 对应的 `close`（与现有行情同步口径一致，一般为**前复权**序列，与项目其它页面一致）。
- **理由**：规格要求「系统内已同步的行情」；项目已有日线主表与同步任务，无需为持仓单独拉行情。
- **备选**：实时行情 / Tushare 即时接口——本期不采用（规格不承诺实时）。

## 2. 整笔盈亏计算方式

- **决策**：对已**清仓**的 `portfolio_trade`，**整笔已实现盈亏**由服务端根据该笔下全部 **`portfolio_operation`** 按**移动加权平均成本**汇总计算（买入类增加持仓与成本，卖出类减少持仓并按当时成本结转损益）；清仓时持仓数量应为 0，** realized_pnl_total** 落库或查询时计算二选一（实现时选「关闭时写入 `realized_pnl` 字段」便于胜率统计与审计）。
- **理由**：可复现、可测试，避免用户手填盈亏与操作明细不一致。
- **备选**：仅用户手填整笔盈亏——验收与对账困难，规格虽允许「直接录入」作为兜底，本期以**自动计算为主**，若计算不可行（数据缺失）再允许管理员式手工覆盖（可选字段，见 data-model）。

## 3. 操作类型与减仓语义

- **决策**：操作类型枚举：`open`（建仓）、`add`（加仓）、`reduce`（减仓）、`close`（清仓）。**减仓**与**清仓**均减少持仓数量；**清仓**要求减至数量为 0 或单独一条 `close` 操作将剩余全部卖出（实现时二选一：**推荐**「最后一次卖出操作标记为 `close` 且数量为减至 0」）。
- **理由**：与规格「建仓—清仓」闭环一致。
- **备选**：仅「买卖」两档——无法区分加仓与建仓语义，不利于列表展示。

## 4. 操作质量自评取值

- **决策**：`operation_rating` 使用 **nullable 枚举**：`good`、`bad`；**未填**表示未评价，**操作胜率**分母仅统计「已评价」操作数，界面展示「已评价数 / 可操作数」。
- **理由**：与规格「不得静默当作坏操作」一致；二分足够 MVP。
- **备选**：三档——可后续扩展，本期不增加。

## 5. 复盘图片本地存储

- **决策**：文件保存在 **`backend/uploads/portfolio/{user_id}/{trade_id}/`**（`trade_id` 为已完结交易主键），文件名 **UUID + 原扩展名**；数据库仅存相对路径或文件名列表；**禁止**路径穿越；MIME 白名单 `image/jpeg`、`image/png`、`image/webp`；单文件上限例如 **5MB**，每笔复盘最多 **9 张**（可在契约中写死，可配置为环境变量）。
- **理由**：规格要求不上 OSS、单机自用；与常见 FastAPI `StaticFiles` 或自定义 `GET /api/portfolio/.../files/{id}` 提供下载/预览一致。
- **备选**：仅存 base64 入 MySQL——占库容大，不采用。

## 6. 与现有认证、路由的集成

- **决策**：新路由挂 `prefix="/api"`，与现有 `auth_router` 一致；所有写操作 **`Depends(get_current_user)`**，按 `user_id` 过滤。
- **理由**：规格要求多用户隔离；项目已具备 JWT + Bearer。

## 7. 定时任务

- **决策**：**本功能不新增定时任务**；参考市值在**读取持仓接口时按需查询**最新日线收盘价即可。
- **理由**：规格未要求定时刷新持仓估值；行情已由现有调度同步日线。
