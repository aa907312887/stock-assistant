# Research：股票基本信息（003）

**功能**: 003-股票基本信息 | **日期**: 2026-03-21

## 1. 周频同步与现有每日全量是否冲突？

**决策**: **并存**。保留 `002` 的每日 17:00 全量同步（含 `stock_basic` + 日线 + 利润表）；**新增**独立周任务，仅执行 `run_sync_basic_only`，只更新 `stock_basic` 及相关静态字段。

**理由**: 规格要求「每周至少一次」面向**基础信息**维度的可观测更新；全量任务已高频更新 `stock_basic` 第一段，但周任务仍满足「独立基础同步」的运维语义与日志关键字，且可在数据源限流紧张时单独调整周任务时间。

**备选**: 取消每日同步中的 `stock_basic` 写入、仅由周任务写入——会改变 `002` 行为，影响综合选股数据新鲜度，**不采纳**。

---

## 2. 无鉴权 API 的暴露范围

**决策**: 后端对 `GET /api/stock/basic`、`POST /api/stock/basic/sync` **不**加 JWT、**不**加 `X-Admin-Secret`；依赖部署网络边界（内网/VPN）降低误暴露风险。

**理由**: 与 `003/spec.md` 假设一致。

**备选**: 与全量手动同步统一走 `ADMIN_SECRET`——与规格「不做权限管理」冲突，**不采纳**。

---

## 3. 前端路由是否必须匿名可访问？

**决策**: **默认**仍将「股票基本信息」页放在需登录的 Layout 下（`requiresAuth: true`），**不在页面内**做角色判断；若产品后续要求整页匿名，再将该路由移出 Layout 并设 `guest: true`。

**理由**: 改动最小；规格禁止的是**权限管理**，不是「是否必须登录应用」。

**备选**: 一步到位匿名路由——需调整产品对「首页/登录」流程的预期，**暂不纳入本功能必选项**。

---

## 4. 与全量同步并发的数据一致性

**决策**: 依赖 MySQL 事务与按 `code` 更新；首版**不加**应用层全局锁；若生产出现锁等待，再在 `tasks.md` 中增加互斥优化。

**理由**: 周任务与日任务同时写 `stock_basic` 概率低；加锁增加死锁风险。

**备选**: 全局 `threading.Lock` 串行两种同步——简单但可能拉长耗时，**作为后续优化**。

---

## 5. 字段映射（Tushare stock_basic 与 ORM）

**决策**: 在 `run_sync_basic_only` 中，将 Tushare 返回的 `area`、`industry`、`list_date` 等映射到 `StockBasic.region`、`industry_name`（及必要时 `industry_code`），与 `docs/数据库设计.md` 对齐；若当前 `run_sync` 仅写 `name/market`，本功能**补齐**静态字段写入。

**理由**: 满足 `003` 关键实体「地域、行业、上市日期」等展示需求。

**备选**: 仅同步 code/name/market——字段不足，**不采纳**。
