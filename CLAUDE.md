# Stock Assistant - Claude Code 项目规则

本文档定义了 stock-assistant 项目的开发规范、工作流程和 AI 助手行为约束。

## 项目概述

stock-assistant 是一个股票筛选与回测系统，包含：
- 后端：Python 3.12 + FastAPI + SQLAlchemy + MySQL + pandas + APScheduler + Tushare API
- 前端：TypeScript + Vue 3 + Vite + Pinia + Element Plus
- 核心功能：综合选股、策略回测、模拟交易、投资逻辑管理

## 核心开发规范

### 1. 文档与注释语言

**规则**：除英文专有名词外，项目文档与代码注释必须使用简体中文编写。

- **适用范围**：
  - 项目文档（`docs/`、`README`、设计说明、Spec 正文等）
  - 代码注释（含 docstring、JSDoc 等）
  - 所有 `specs/` 目录下的规格文档

- **专有名词例外**：库名、API 名、协议名、标准缩写（如 HTTP、JSON）、品牌与产品名等可保留英文

- **禁止**：在注释或文档中用英文写完整说明句（仅因「习惯英文」而不用中文）

### 2. Git 工作流

**规则**：在主分支上直接开发并推送，不新建功能分支。

- **在主分支上直接开发**：默认主分支为 `main`（若仓库使用 `master` 则指 `master`）。所有功能与修复均在该分支上提交，**不要新建功能分支**。
- **直接推送到远程**：开发完成后执行 `git push origin main`（或 `git push origin master`）推送到远程，无需开 PR、无需合并其他分支。
- **不主动建议**：创建新分支、切换分支、merge 其他分支、PR 流程等；若用户明确要求分支或 PR 再另行处理。

### 3. Agent 与 Git 写入操作

**规则**：未经用户明确同意，助手不得执行会改变本地或远程仓库状态的 Git 命令。

- **禁止擅自提交/推送**：未经用户**明确同意**，助手不得执行 `git commit`、`git push`、`git merge`、`git rebase`、`git cherry-pick`、`git reset --hard` 等会**改变本地或远程仓库状态**的 Git 命令。
- **允许只读**：可执行 `git status`、`git diff`、`git log`、`git show` 等用于排查与说明的只读命令。
- **视为同意**：用户明确说出「提交」「commit」「推送」「push」「推到远程」等指令时，再按用户意图执行对应操作。

### 4. Spec 驱动开发

**规则**：代码变更需同步更新对应 spec.md。

- **规范来源**：本项目按 **Spec 模式**开发；功能需求与行为约定以 `specs/<功能目录>/` 下的规范为准。
- **同步 `spec.md`**：实现或修改与某功能相关的代码、接口或数据模型时，**必须同步更新**该功能对应的 `specs/**/spec.md`，使规范与实现一致；若变更影响范围、数据模型或 API 契约，还应视情况更新同目录下的 `plan.md`、`data-model.md`、`contracts/` 等已有工件。
- **新建功能**：新增能力时应在 `specs/` 下建立（或沿用）功能目录，并维护完整的 `spec.md` 与相关文档，再按规范实现代码。

### 5. 策略类注释规范（强制）

**规则**：策略相关类必须写清晰详细的中文注释（便于回顾操作现场/回测复现）。

**适用范围**：后端所有 **策略相关类**（包括策略接口/抽象基类、具体策略实现类、与策略强绑定的判定器/过滤器/信号生成器等）。

#### 总原则

- 注释必须使用**简体中文**，且要**详细清楚**，能让后来者在不读完整实现代码的情况下理解策略意图与口径。
- 注释要覆盖"为什么这么做"和"口径是什么"，避免只写"做了什么"。
- 当策略逻辑/口径变更时，必须**同步更新注释**，否则视为未完成。

#### 最低要求（每个策略实现类必须包含）

在类的 docstring（或等效注释块）中，至少写清以下内容：

1. **策略名称与目标**：一句话说明策略想捕捉的市场行为/形态。
2. **适用范围**：市场（如 A 股）、数据粒度（日/周/月）、必要前置数据字段（如 open/high/close/prev_close）。
3. **核心规则（分步）**：
   - 触发条件（如何定义"大涨/大阳线"等）
   - 过滤条件（最近 N 日约束等）
   - 买入/卖出判定（若本期仅筛选，也要说明未来回测如何使用这些方法）
4. **关键口径与阈值**：用公式写清百分比口径（例如 \((high-open)/open\)），并列出阈值来源与含义。
5. **边界与异常**：空数据、缺失字段、非交易日、数据尚未同步等应如何处理/如何判定不可用。
6. **输出与可追溯性**：策略会产出哪些"操作现场"记录（例如执行快照、信号事件、候选明细），各自用于什么目的。
7. **示例（至少 1 个）**：用 2～3 个交易日的简化例子描述"满足/不满足"的典型情况，帮助校验口径。

### 6. 前端产品能力提示

**规则**：前端页面须提供悬浮能力说明，不占用大面积版面。

- **触发时机**：本页涉及的产品能力**新增**或**变更**时，必须在**该页**补充或更新能力说明；与 `specs/**/spec.md` 等规格保持一致，避免页面与规格口径漂移。
- **展示形式**：以**悬浮展示**为主（鼠标移入即显、移出即收），例如 Tooltip、轻量 Popover、带样式的 `title` 等；**不得**用大块固定文案占满首屏或挤压主内容区。
- **文案内容**：用**简短中文**说明「本页能做什么、不做什么或边界」（可与页面标题/区块标题旁的提示图标配合）；避免堆砌实现细节或内部字段名。
- **交互与占位**：提示控件应小（如标题行末的「?」图标或信息图标），默认**不展开**；移动端若无 hover，可采用**轻触展开**的同等信息层级，仍避免大面积常驻说明。
- **可访问性**：在实现悬浮提示时，尽量保证键盘或读屏可用（视所选组件能力而定），不强制但与 UI 库最佳实践对齐。

## 技术方案详细度要求

技术方案必须达到**可直接按方案实现**的粒度，禁止仅写概要或一句话带过。

### 定时任务设计要求

若功能涉及定时任务（如每日拉数、定时同步），必须明确写清：

1. **使用什么组件**（如 APScheduler、cron、Celery 等）及在项目中的位置
2. **如何注册**（在应用启动的何处、何时注册，如 FastAPI lifespan 内）
3. **调度策略**（cron 表达式或具体时间，如「每日 17:00」）
4. **部署时是否执行一次**（是/否，若「是」写清触发方式：应用启动后延迟 N 秒、或单独的管理接口/脚本）
5. **是否提供手动触发方式**（HTTP 接口路径与方法、或管理命令、或脚本命令，以及鉴权方式）
6. **失败与重试**（失败是否重试、重试次数、日志与告警）

### 接口与数据流

写清请求路径、主要参数、响应结构、错误码或错误信息约定；数据从哪来到哪去、经哪些层。

### 数据模型

表/实体、主键、关键字段与类型、索引、与现有表的关系。

### 前端与后端职责

哪些逻辑在后端、哪些在前端，关键接口与页面一一对应。

## Spec Kit 工作流（参考）

本项目原使用 Cursor 的 Spec Kit 命令进行规格驱动开发。以下是核心工作流程供参考：

### 核心命令流程

1. **speckit.constitution** - 设定项目宪法/原则
2. **speckit.specify** - 根据描述创建或更新功能规格
3. **speckit.clarify** - 澄清规格中的歧义
4. **speckit.plan** - 生成技术实现计划
5. **speckit.tasks** - 拆解为可执行任务
6. **speckit.implement** - 按任务执行实现
7. **speckit.analyze** - 跨制品一致性分析（可选）
8. **speckit.checklist** - 生成质量检查清单（可选）

### 关键原则

- **Spec 优先**：所有功能开发前先完成 spec.md
- **中文为主**：所有规格文档、技术方案、任务列表均使用中文
- **可测试性**：每个需求必须可测试、可验证
- **追溯性**：代码变更必须能追溯到对应的 spec 和 plan

## 注意事项

1. **不要**在未读取文件的情况下修改代码
2. **不要**创建不必要的新文件，优先编辑现有文件
3. **不要**添加未被要求的功能或重构
4. **必须**在策略类中写详细的中文注释
5. **必须**同步更新 spec.md 与代码实现
6. **必须**在前端页面添加悬浮能力说明
7. **必须**使用中文编写所有文档和注释（专有名词除外）
